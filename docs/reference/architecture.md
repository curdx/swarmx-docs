# 工作原理

本文从概念层面说明 swarmx 如何运行你的 CLI、如何让多个 agent 协作，以及它对隐私与安全的处理方式。

## 定位

swarmx 在伪终端中原样运行你已安装的 claude、codex、opencode、reasonix 二进制，通过共享收件箱与黑板让它们协作完成同一任务。它不是又一个模型客户端——鉴权、限流与套餐额度的行为，与你在终端中直接使用完全一致。

## 三层结构

swarmx 自下而上分为三层，各层职责单一：

| 层 | 职责 |
|---|---|
| 终端层 | 未经修改的 CLI 二进制运行于 PTY 之上。鉴权、限流与用量行为与终端环境一致，凭证始终由 CLI 自身管理。 |
| 适配层 | 一个极薄的包装进程负责启动 CLI，并向服务端上报「已就绪」与「已退出」两个生命周期信号。 |
| 工具层 | 协作能力以原生 MCP 工具的形式暴露给模型：发送消息、读写黑板、派发 worker 等。模型像调用任何其他工具一样调用它们。 |

在此之上，一个基于 Rust 的服务端（仅监听本地回环地址）统筹进程生命周期、收件箱、黑板、会话录制与唤醒调度。

## 数据流

以创建一个 agent 为例：

1. 浏览器向服务端请求创建 agent，指定使用的 CLI 与工作目录。
2. 服务端为该 agent 写入 MCP 与钩子配置，随后在 PTY 中启动 CLI。
3. CLI 就绪后，服务端向所有前端广播状态，并开始将终端字节流录制为可回放的会话文件。
4. 浏览器为该 agent 建立一条 WebSocket 实时终端连接；另有一条事件流持续接收状态与消息更新。
5. 运行中的模型通过 MCP 工具发送消息、读写黑板，这些调用最终由服务端处理并持久化。

## 唤醒机制

编码 CLI 通常是「处理完一个回合即停下等待」的模式。这意味着即便另一个 agent 向它发送了消息，它也不会主动醒来。swarmx 通过两条路径解决：

- **回合结束时检查**：CLI 每结束一个回合，会触发一次检查。若存在未读消息，该 agent 被要求续跑一个回合去读取并回应；否则正常停止。
- **写入即唤醒**：当某个黑板键被写入时，服务端立即唤醒所有声明依赖该键的 agent——包括此前已停下的。整个过程无需轮询，也不会因互相等待而死锁。

## 黑板与收件箱

两种协作载体各司其职：

| 载体 | 承载内容 | 特点 |
|---|---|---|
| 黑板 | 结构化产物与共享上下文 | 多对多、键值、持久，支持全文检索与版本历史；写入自动唤醒订阅者。 |
| 收件箱 | 「发生了某事」的点对点信号 | 一对一、有序、可追踪已读状态；消息触发接收方的回合检查。 |

典型模式是：将产物写入黑板，再发一条通知消息。订阅者由黑板写入唤醒，未订阅的接收方由消息唤醒。

## 各引擎差异

不同 CLI 的驱动方式不同，但仪表盘对外呈现的是统一的 agent 视图：

| 引擎 | 运行方式 | 自动唤醒 |
|---|---|---|
| claude | PTY 终端 | 支持 |
| codex | PTY 终端 | 支持（需 0.132 及以上） |
| opencode | 全屏 TUI，经其官方控制接口驱动 | 支持 |
| reasonix | HTTP / SSE 服务 | 支持 |

## 隐私与安全边界

- 服务端仅绑定 `127.0.0.1`，不对外暴露；中间件拒绝所有非本地来源的请求。
- swarmx 不读取、也不存储任何登录凭证——它只将 `HOME` 透传给子进程，复用 CLI 已有的登录状态。
- 启动子进程时从空白环境构建变量，避免将敏感变量泄漏给 CLI。

## 打包与运行时资源

桌面版通过 Tauri 打包，服务端及其配套二进制以 sidecar 形式内嵌，所需的内置资源在编译期一并打入。因此用户下载、安装、打开即可使用，不依赖任何仓库目录或手动配置。运行参数可通过环境变量调整，详见[配置项](/reference/configuration)。
