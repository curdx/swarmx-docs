# 引擎参考

swarmx 支持五个引擎。它们的运行方式不同，但都遵循同一条凭证原则：swarmx 启动真实二进制、把 `HOME` 透传给它，**从不读取或存储任何 token**，直接复用你已有的登录状态。安装与登录见[安装与登录引擎](/guide/install-engines)。

## 自动唤醒对照

| 引擎 | 运行方式 | 鉴权 | 自动唤醒 |
|---|---|---|---|
| **claude** | 终端界面，键入驱动 | 复用 `~/.claude` 订阅登录 | 支持（回合末钩子） |
| **codex** | 终端界面，键入驱动 | 复用 `~/.codex` 登录 | 支持，需 **0.132+**；更低版本休眠 |
| **opencode** | 全屏终端界面，经官方控制接口驱动 | 按供应商登录 | 支持（空闲事件插件），需内置插件可定位 |
| **reasonix** | 无界面 HTTP/SSE 服务 | `DEEPSEEK_API_KEY` | 支持（回合结束事件，无需钩子） |
| **zulu** | 无界面 HTTP/SSE 服务 | Comate License | 支持（回合结束事件，无需钩子） |

## claude

以全屏终端界面运行，通过键入驱动。透传 `HOME` 后复用 `~/.claude` 里的订阅登录，并刻意停留在**交互订阅**这一计费面上（屏蔽 `ANTHROPIC_` 开头的环境变量，避免被切换到 API 计费）。自动唤醒通过回合末的检查钩子实现。模型层级用 Claude 自己的 opus/sonnet/haiku，并支持思考强度。

**适合作** 编排器（队长）或通用 worker，是最低要求的引擎。

## codex

同样以终端界面、键入方式运行。透传 `HOME` 后复用 `~/.codex`（`codex login` 之后）。

自动唤醒需要 **codex 0.132 及以上**——较新的 codex 才具备让外部回合末钩子免逐次信任的能力，swarmx 会自动探测并启用；在更低版本上引擎照常运行，但自动唤醒处于休眠状态，直到手动信任。模型不使用 opus/sonnet/haiku 层级（它是 gpt-5.x 系）。

**适合作** 编排器或 worker。

## opencode

以全屏终端界面运行，但提示词通过它**官方的控制接口**下发，而非模拟键入（大段提示用键入粘贴会卡住不提交，故走接口）。按供应商登录（`opencode auth login`），登录状态共享。

自动唤醒通过一个在**空闲事件**上触发的插件实现——需要 swarmx 内置的插件文件能被定位（桌面版由应用注入其打包路径）。若定位不到，worker 仍能使用协作工具，但**不具备自动唤醒**，功能降级而不崩溃。

**适合作** 编排器或 worker。

## reasonix

不是终端界面，而是以 `reasonix serve` 起一个无界面的 HTTP/SSE 服务，swarmx 通过 HTTP 提交每一回合、通过事件流接收回合结束与工具动作。通过 `DEEPSEEK_API_KEY` 使用 DeepSeek，内置 flash / pro 两档。因无终端画面，界面默认展示结构化的「活动」视图。自动唤醒由回合结束事件驱动，无需钩子。

**适合作** worker。

## zulu

同样是无界面的 HTTP/SSE 服务（`zulu serve`），用一份 **Comate License** 认证。它的关键特性是**模型按请求指定**：同一个服务进程可以服务任意一个模型，因此一份 License 就能让十余个模型并行——这正是[研究委员会](/guide/research-committee)与[融合竞赛](/guide/fusion)的引擎。

为支持多个 zulu agent 并发，swarmx 为每个 agent 分配独立的运行环境，避免它们的后台服务互相抢占端口。License 认证与此无关，因此隔离环境不影响登录。自动唤醒由回合结束事件驱动。

**适合作** 多模型协作的引擎，也可作通用 worker。

::: tip 引擎与角色
哪个引擎适合当队长、哪个适合当 worker，源自它们的运行方式（有终端画面的更适合当有「驾驶舱」的队长，无界面的更适合当 worker 或委员会成员），而非硬性限制。角色与引擎解耦，见[编排器与角色](/guide/orchestrator-and-roles)。
:::
