# 融合竞赛

融合竞赛将**同一个需求**交由多个模型分别实现，每个模型在隔离的 git worktree 中独立完成；可附加一条**客观检查命令**（如 `pytest`、`cargo test`）作为硬性门禁，再由评审模型综合各家最优方案并合并回主线。

## 一键全自动

只需输入一句需求、勾选「全自动」并点击「开赛」，后续流程自动完成：

1. 服务端自动挑选可用模型组成参赛组合（存在两个及以上不同 CLI 时跨引擎对赛，否则以一份 Zulu License 运行多个模型）。
2. 每个模型在各自隔离的 worktree 中实现需求。
3. 参赛模型全部提交后，自动进入综合评审阶段。
4. 评审模型综合出最优版本并合并至主线。

全程无需人工干预。裁决由**服务端守护机制**兜底：无论模型是否正常收尾，批次都不会静默卡死——达到条件即合并，否则退回人工裁决。

## 手动模式

关闭「全自动」即为分步流程：自行指定 2 至 4 个参赛模型、驱动其实现，再点击「评审」或「综合最优」，最终选定优胜方案并合并。适用于需要掌控每一步的场景。

## 客观检查（可选，推荐启用）

提供一条检查命令（如 `python3 check.py`、`cargo test`）后，swarmx 会在每个参赛模型的 worktree 中真实执行。相比人工阅读代码差异，这是更硬的证据：任何未通过检查的方案都不会当选，无论其代码看起来多么优雅。

## 前置条件

多模型能力由 Comate Zulu 提供（跨引擎模式下则使用你已登录的 claude、codex 等）。配置方式见[安装 Zulu 与配置 License](/guide/zulu-setup)。
